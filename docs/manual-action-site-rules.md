# 手动处理状态与网站规则方案

## 背景

部分网站不会用失败状态响应自动访问。以微信公众号文章为例，请求会先跳转到验证页，验证页本身仍返回成功状态。保存工具因此可能把“环境异常，请完成验证”的页面当成有效文章保存，并把任务标记为成功。

系统已有“需登录”状态，但它只面向视频下载。随着微信公众号验证以及未来其他网站的登录、验证码、扫码确认等情况出现，继续增加彼此平行的状态会让任务状态、页面按钮和恢复流程越来越难维护。

## 目标

1. 将“需登录”“需验证”等情况统一为“需手动处理”。
2. 任务状态只表达是否需要用户介入，具体原因和下一步操作由结构化的手动处理事项表达。
3. 网站特殊判断采用独立规则模块，新增网站时不修改主保存流程。
4. 微信验证页不能再被当作有效文章、进入搜索或覆盖原文章标题。
5. 保留网页、视频并行保存和部分成功能力，一条任务可以同时存在多个待处理事项。
6. 旧数据自动兼容，用户升级镜像后直接生效，不要求修改 compose 文件或增加配置。

## 非目标

1. 不尝试绕过网站验证码或风控。
2. 不尝试把一个标签页中的验证结果复制到另一个标签页；系统直接保存用户完成操作的原标签页。
3. 不用网站标题、单个文字或最终 HTTP 状态作为唯一判断依据。

## 统一状态

任务增加统一状态 `manual_action_required`，界面显示“需手动处理”。原有 `browser_login_required` 不再用于新任务，升级时转换为统一状态。

每个待处理事项包含：

- 稳定编号：用于去重、恢复和测试。
- 类型：登录、验证或其他人工确认。
- 影响对象：网页或视频。
- 用户提示：说明发生了什么。
- 恢复方式：只重试网页或继续下载视频。
- 规则编号：记录由哪个网站或工具规则产生。

任务允许保存多个待处理事项。例如网页需要微信验证、视频又需要登录时，两项都保留，界面分别展示。

状态变化为：

`排队 -> 处理中 -> 需手动处理 -> 处理中 -> 已完成 / 失败 / 需手动处理`

“需手动处理”不是失败，任务完成时间保持为空。服务重启后，该状态和待处理事项必须保留。

## 网站规则体系

规则体系由规则登记处和独立网站规则组成。

每条规则只负责：

1. 判断网址是否属于自己。
2. 读取已经保存的页面并检查稳定特征。
3. 返回以下标准结果之一：正常、需手动处理、内容不可用。

规则不得修改正常存档，也不得直接修改任务、删除文件、创建、匹配或控制标签页。主保存流程统一管理任务、文件和标签页生命周期，网站规则不参与判断两个页面是否属于同一任务。

规则按明确域名匹配，不允许用包含关系匹配整个大域名。规则检查只读取有限大小的文件内容，不执行页面脚本，不记录验证码、Cookie、完整查询参数等敏感信息。

如果某条已匹配的网站规则自身出错，任务应当报告“无法确认存档有效性”，不能继续误报成功。

## 微信公众号规则

第一版只匹配 `mp.weixin.qq.com`。

### 需验证

必须同时满足多个高可信特征：

- 可见文字包含“环境异常”。
- 可见文字包含“完成验证后即可继续访问”。
- 页面存在“去验证”控件或腾讯验证码组件。

HTTP 200、`retkey: 14`、`Weixin Official Accounts Platform` 标题都不能单独触发规则。正常微信文章也可能使用该英文标题。

命中后返回一个网页验证事项：

- 类型：验证。
- 影响对象：网页。
- 提示：微信要求完成访问验证，正文尚未保存。
- 恢复方式：完成验证后重新归档。

### 内容不可用

如果页面明确显示“此内容暂时无法查看”，返回内容不可用。它属于失败而不是手动处理，因为用户完成验证也不能保证恢复内容。

### 正常文章

只要没有命中上述高可信组合，就不因英文平台标题、微信样式或普通文章文字而误判。
正常文章与所有其他网站使用完全相同的保存流程。微信规则不再重新抓取图片、补充样式或改写保存后的文件。

## 主保存流程

1. 主流程先创建一个浏览器标签页，立即把浏览器返回的唯一标识与任务绑定，再让保存工具在这个标签页中打开并保存网页。
2. 规则登记处按原网址选择适用规则。
3. 规则检查保存结果。
4. 正常结果直接提取标题并进入搜索处理，不再执行网站专属改写。
5. 需手动处理时删除无效验证页，不更新标题，不进入搜索，并记录待处理事项。
6. 内容不可用时删除无效页面并按网页保存失败处理。
7. 网页与视频结果汇总：任何待处理事项存在时，任务进入“需手动处理”；已经成功的有效文件继续保留。
8. 网页被判断为需手动处理时保留原标签页。用户完成操作后，保存工具直接保存这个标签页，不重新访问网址；重试网页时保留已经下载成功的视频，继续视频时保留已经保存成功的网页。
9. 重试后仍命中规则时，任务继续保持“需手动处理”，不得误报成功。
10. 成功或终止时只关闭 Reader 自己创建的临时标签页。

## 动态页面存档完整性

- 保存流程不得用固定等待秒数猜测页面是否完成，也不得通过不断增加等待时间掩盖资源缺失。
- SingleFile 只在浏览器明确报告页面网络活动结束后开始保存；如果在加载上限内始终没有完成信号，本次保存失败，不降级为较早的页面事件继续保存。
- 不给所有页面请求强加 `Cache-Control: no-cache` 或 `Pragma: no-cache`。这类请求改写会改变部分网站的资源加载行为，已经确认会让哔哩哔哩主体排版缺失。
- 关闭 SingleFile 的“删除未使用样式”和“删除其他屏幕样式”精简。现代网页会在运行过程中组合样式，保存时误删会造成标题、分栏、背景和播放器布局损坏。
- 延迟图片仍使用 SingleFile 的滚动触发与资源空闲判断；最大时间只作为失败上限，不作为固定睡眠。
- 普通网页、需手动处理后保存的当前标签页和命令行诊断脚本使用同一套保存设置。

## 标签页绑定与丢失

- 标签页绑定属于统一浏览器流程，对所有网站和所有被识别出的手动处理页生效，不是微信专用逻辑。
- 网页和视频分别保存绑定，两个待处理事项不能占用同一个标签页。
- 正常操作始终只使用保存的唯一标识。即使标签页内发生验证跳转、网址变化或标题变化，也仍然是同一个标签页。
- 切回和继续处理都只查找这个唯一标识，不按标签页顺序、网址、标题、网站规则或页面内容匹配。
- 服务或浏览器重启后，如果原唯一标识仍然存在就继续使用；标识失效则直接标记为“原标签页已丢失”，不猜测其他页面。
- 网址只作为任务记录信息，不参与标签页判断；不保存账号、密码或 Cookie。
- 原标签页关闭后，任务继续保持“需手动处理”。只有用户明确点击“重新打开处理页面”才会创建新标签页。
- 同一个标签页即使被用户打开成其他内容也仍视为原标签页，因此页面提示用户在继续前确认当前显示的是要处理的内容。

## 页面行为

- 列表和详情统一显示“需手动处理”。
- 每项待处理事项显示自己的原因和影响对象。
- 每项待处理事项显示自己的标签页状态，并分别提供“打开处理页面”“切回处理页面”或“重新打开处理页面”。
- 网页验证事项提供“继续处理”，没有可用标签页时不能执行。
- 视频登录事项提供“登录后继续下载”。
- 网页和视频的状态图标分别显示各自是否等待人工操作。
- 无效验证页不出现在文件列表；已经有效保存的其他文件仍可查看。

## 兼容与升级

- 数据升级新增待处理事项存储，并将旧 `browser_login_required` 任务转换成 `manual_action_required`。
- 旧任务的原因转换为视频登录，恢复方式保持“继续下载视频”。
- 升级时清理历史错误文字和待处理提示中的临时令牌、签名等查询值；新错误在写入前清理，页面显示时再次兜底。
- 旧的继续下载入口暂时保留，避免已有调用失效。
- 旧的任务级浏览器入口暂时保留；只有一个待处理事项时使用对应标签页，多个事项时要求调用方指定具体事项。
- 升级前已经等待处理的任务没有可恢复的标签页，升级后标记为“原标签页已丢失”，只能由用户明确重新打开一次。
- 新任务只产生统一状态。
- 浏览器连接和指定标签页保存能力随镜像提供默认配置，不依赖新增环境变量或 compose 参数。

## 测试范围

### 规则测试

1. 精简后的真实微信验证页能够命中。
2. 使用英文平台标题的正常微信文章不能命中。
3. 普通网站出现相同文字不能命中。
4. “此内容暂时无法查看”被识别为内容不可用。
5. 不完整、超大或异常页面不会造成误报或失控读取。
6. 正常微信文章保存后内容保持原样，不被网站规则二次改写。

### 保存流程测试

1. 微信验证页进入“需手动处理”，无效文件被删除。
2. 验证页标题不覆盖原文章标题。
3. 验证页不进入搜索处理。
4. 完成验证后保存原标签页成功，任务进入已完成，期间不创建第二个相同网址标签页。
5. 重试仍需验证时，任务保持“需手动处理”。
6. 视频登录继续使用统一状态，并能继续下载。
7. 网页验证和视频登录可以同时存在。
8. 有效网页或视频部分结果不会因另一部分需要人工处理而丢失。
9. 服务重启后待处理事项不丢失。
10. 旧状态升级后能够继续操作。
11. 标签页关闭后继续操作会停留在手动处理状态，明确重新打开后才能继续。
12. 多个相同网址不会被自动认领，多个任务以及网页和视频同时待处理时只使用各自记录的唯一标识。

### 交付验证

1. 后端全部测试通过。
2. 前端接口生成、检查、格式检查、构建和单元测试通过。
3. Docker 镜像重新构建并启动成功。
4. 本地页面实际走通“识别 -> 提示 -> 切回原标签页 -> 继续处理”流程，并核对标签页数量不增加。
5. 用正常微信文章确认全站通用流程能够保存主图、后续图片、正文宽度、标题和作者信息。
6. 确认微信存档不经过额外图片下载或排版改写，断网后仍可阅读。
7. 用哔哩哔哩视频页确认标题、播放器区域、右侧信息栏和页面背景保持原有排版，并确认流程没有固定等待参数。

## 完成标准

- 微信验证页不再显示为已完成。
- 正常微信文章不被误判。
- 登录、验证统一显示为“需手动处理”，但操作提示准确。
- 一条任务可以携带多个待处理事项。
- 新增网站规则不需要修改主保存流程。
- 旧数据、旧操作入口和现有网页/视频保存行为保持兼容。
- 无需用户修改部署配置。
- 正常微信文章完全依靠全站通用流程保存，网站规则不改写图片和排版。
- 微信正文宽度、标题、作者信息和图片排版与原网页接近，断网后仍可阅读。
- 所有自动检查和本地实际验证完成。
- 用户完成验证后保存的是刚才操作的标签页，全程不出现第二个相同网址标签页。
